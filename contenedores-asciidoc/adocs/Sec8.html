Sección 8. Instalación de k3s sobre Linux

El objetivo del presente capítulo es desplegar un clúster altamente disponible de k3s.

Acerca de K3s

K3s es una distribución de kubernetes caracterizada por ser liviana, sencilla y completa, en el sentido de que ya nos brinda de antemano varios elementos que estaríamos necesitando en cualquier instalación de Kubernetes:

  • Ingress Controller: el controlador de ingreso se encarga de recibir las peticiones externas al clúster y canalizarlas hacia las aplicaciones/contenedores internos a través de la definición de reglas de ingreso.

  • Storage Provisioner: el aprovisionador de almacenamiento se encarga de que automáticamente cuando se define un recurso de tipo PersistentVolumeClaim (Reclamación de Volumen Persistente), se generen automáticamente los recursos internos necesarios para que funcione la capa de persistencia de cara al usuario de Kubernetes, es decir, se genera el recurso PersistentVolume (Volumen Persistente) asociado y el espacio físico del volumen asociado, por ejemplo un subdirectorio en un directorio compartido por NFS.

  • Networking Provider: el proveedor de red se encarga de asignar una IP a cada Pod y crear túneles de red entre los distintos nodos de Kubernetes, creando así efectivamente una red definida por software (SDN, Software Defined Networking) programática que permita gestionar las abstracciones de red de k8s como ser los Services (servicios) o NetworkPolicies (políticas de red)

Configuración de HAProxy y NFS server

Para la primera parte del curso, necesitamos configurar el balanceador de carga y el storage NFS para utilizarlos a lo largo del mismo.

El servidor HAProxy será el balanceador de capa 4 responsable por todas las interacciones externas al clúster, ya sea a través de la API de Kubernetes (puerto 6443/tcp) así como por el ingress controller (puertos 80 y 443/tcp)

Note
 Los siguientes pasos se deben ejecutar con el usuario “root” en el servidor “lb”

  1. Configurar selinux y firewalld

    firewall-cmd --permanent --add-port=6443/tcp
firewall-cmd --permanent --add-service=http --add-service=https --add-service=nfs
firewall-cmd --reload
setsebool -P httpd_can_network_connect 1
semanage port -a -t http_port_t 6443 -p tcp

  2. Instalar haproxy y el editor de texto vim

    dnf install -y haproxy vim

  3. Configuración del archivo haproxy.cfg

    Important
    		 se debe reemplazar las instancias de “example.com” por el nombre de dominio DNS asignado para el alumno. 
    vim /etc/haproxy/haproxy.cfg
    #---------------------------------------------------------------------
# Example configuration for a possible web application.  See the
# full configuration options online.
#
#   https://www.haproxy.org/download/1.8/doc/configuration.txt
#
#---------------------------------------------------------------------

#---------------------------------------------------------------------
# Global settings
#---------------------------------------------------------------------
global
    # to have these messages end up in /var/log/haproxy.log you will
    # need to:
    #
    # 1) configure syslog to accept network log events.  This is done
    #    by adding the '-r' option to the SYSLOGD_OPTIONS in
    #    /etc/sysconfig/syslog
    #
    # 2) configure local2 events to go to the /var/log/haproxy.log
    #   file. A line like the following can be added to
    #   /etc/sysconfig/syslog
    #
    #    local2.*                       /var/log/haproxy.log
    #
    log         127.0.0.1 local2

    chroot      /var/lib/haproxy
    pidfile     /var/run/haproxy.pid
    maxconn     4000
    user        haproxy
    group       haproxy
    daemon

    # turn on stats unix socket
    stats socket /var/lib/haproxy/stats

    # utilize system-wide crypto-policies
    ssl-default-bind-ciphers PROFILE=SYSTEM
    ssl-default-server-ciphers PROFILE=SYSTEM

#---------------------------------------------------------------------
# common defaults that all the 'listen' and 'backend' sections will
# use if not designated in their block
#---------------------------------------------------------------------
defaults
    mode                    tcp
    log                     global
    option                  dontlognull
    option http-server-close
    option                  redispatch
    retries                 3
    timeout queue           1m
    timeout connect         10s
    timeout client          1m
    timeout server          1m
    timeout http-keep-alive 10s
    timeout check           10s
    maxconn                 3000

frontend api
    bind *:6443

    default_backend             api

    balance     roundrobin
    server  m1 m1.s1.example.com:6443 check
    server  m2 m2.s1.example.com:6443 check
    server  m3 m3.s1.example.com:6443 check

frontend http
    bind *:80

    default_backend             http

backend http
    balance     roundrobin
    server  m1 m1.s1.example.com:80 check
    server  m2 m2.s1.example.com:80 check
    server  m3 m3.s1.example.com:80 check
    server  w1 w1.s1.example.com:80 check
    server  w2 w2.s1.example.com:80 check

frontend https
    bind *:443

    default_backend             https

backend https
    balance     roundrobin
    server  m1 m1.s1.example.com:443 check
    server  m2 m2.s1.example.com:443 check
    server  m3 m3.s1.example.com:443 check
    server  w1 w1.s1.example.com:443 check
    server  w2 w2.s1.example.com:443 check

  4. Iniciar el servicio de haproxy

    systemctl enable --now haproxy

  5. Instalar el paquete de NFS y levantar el servicio asociado

    dnf install -y nfs-utils
systemctl enable --now nfs-server

  6. Configurar el punto de exportación

    mkdir /shared
chmod 777 /shared
echo '/shared *(rw,sync)' >> /etc/exports.d/k8s.exports
exportfs -rv

Despliegue de k3s configurado por defecto

https://docs.k3s.io/datastore/ha-embedded

Warning
 La configuración por defecto de k3s utiliza un overlay VXLAN con el plugin de red llamado Flannel, el cual podría interferir con otras redes VXLAN a nivel de hipervisor físico, como por ejemplo NSX y NSX-T de VMWare, o Neutron de OpenStack. Para desplegarlo con el backend de red wireguard-native, se puede seguir el procedimiento Re-instalación de k3s con proveedor de red wireguard-native

El despliegue de k3s en alta disponibilidad requiere de la ejecución de los siguientes comandos en el orden especificado, utiliza la base de datos etcd embebida en los nodos del plano de control para guardar la configuración del clúster.

Note
 Los siguientes comandos deben ser ejecutados con usuario root en los servidores indicados

  1. Configurar firewall en cada servidor master (m1, m2, m3)

    firewall-cmd --permanent --add-port=2379/tcp --add-port=2380/tcp --add-port=6443/tcp \
            --add-port=8472/udp --add-port=10250/tcp
firewall-cmd --permanent --add-service=http --add-service=https
firewall-cmd --permanent --zone=trusted --add-source=10.42.0.0/16 --add-source=10.43.0.0/16
firewall-cmd --reload

  2. Configurar firewall en cada servidor worker (w1, w2)

    firewall-cmd --permanent --add-port=8472/udp --add-port=10250/tcp
firewall-cmd --permanent --add-service=http --add-service=https
firewall-cmd --permanent --zone=trusted --add-source=10.42.0.0/16 --add-source=10.43.0.0/16
firewall-cmd --reload

  3. Despliegue del primer servidor (m1)

    El parámetro --cluster-init en el primer nodo master, indica al script de instalación de k3s que debe inicializar un plano de control altamente disponible basado en etcd, en lugar del backend por defecto basado en SQLite, utilizado más para pruebas en entornos no productivos.

    El parámetro --tls-san apunta al DNS del balanceador de carga para agregarlo a la propiedad SubjectAlternativeNames (SAN) del certificado TLS de la API

    curl -sfL https://get.k3s.io | K3S_TOKEN=SECRET sh -s - server \
    --cluster-init --tls-san api.k8s.s1.example.com

  4. Despliegue del segundo y tercer servidor (m2, m3)

    El parámetro --server indica al instalador de k3s que la API debe apuntar al balanceador de carga HAProxy que hemos configurado de antemano. Por otro lado, el argumento server indica que este nodo debe contener también la base de datos etcd y la API de Kubernetes, cumpliendo efectivamente con el rol de un nodo de plano de control.

    curl -sfL https://get.k3s.io | K3S_TOKEN=SECRET sh -s - server \
    --server https://api.k8s.s1.example.com:6443 --tls-san api.k8s.s1.example.com

  5. Despliegue de los agents (workers w1, w2)

    El argumento agent indica que estos nodos cumplirán el rol de nodos worker de kubernetes, es decir, solamente albergarán contenedores de aplicativos desplegados sobre la plataforma, no así otros componentes internos como etcd, los controllers, scheduler, o apiserver que corren en los nodos de plano de control.

    curl -sfL https://get.k3s.io | K3S_TOKEN=SECRET sh -s - agent --server https://api.k8s.s1.example.com:6443

  6. Verificación básica de la instalación (m1)

    kubectl get nodes
    Resultado esperado
    [root@m1 ~]# kubectl get nodes
NAME              STATUS   ROLES                       AGE     VERSION
m1.s1.example.com   Ready    control-plane,etcd,master   10m     v1.28.4+k3s2
m2.s1.example.com   Ready    control-plane,etcd,master   8m6s    v1.28.4+k3s2
m3.s1.example.com   Ready    control-plane,etcd,master   6m38s   v1.28.4+k3s2
w1.s1.example.com   Ready    <none>                      3m      v1.28.4+k3s2
w2.s1.example.com   Ready    <none>                      2m24s   v1.28.4+k3s2
[root@m1 ~]#

El resultado de estos comandos es que ahora tenemos un clúster altamente disponible instalado y listo para ser utilizado

Recordemos que gracias al balanceador de carga HAProxy ahora tenemos la posibilidad de apuntar a la API a través de la URL https://api.k8s.s1.example.com:6443 y por debajo estaremos contactando con uno de los nodos master.

Instalación de herramientas de administración

El objetivo de esta sección es que el alumno sea capaz de utilizar herramientas de administración de kubernetes tanto desde la máquina virtual m1 correspondiente al primer master de la instalación de k3s, así como también desde su propia máquina de trabajo. Se asume que el alumno cuenta con acceso como Administrador a una máquina de trabajo con sistema operativo Microsoft Windows ™

Existen muchas herramientas que nos pueden ayudar a administrar un clúster de kubernetes, en este curso nos estaremos enfocando en las siguientes:

  1. Kubectl: herramienta de línea de comando por excelencia para adminsitrar los clústeres de kubernetes

  2. OpenLens: herramienta gráfica gratuita y open source perteneciente a la empresa Mirantis, tiene una versión de pago llamada KubeLens.

  3. Helm: gestor de paquetes por excelencia para Kubernetes, se utiliza también desde la línea de comandos así como kubectl.

  4. Kustomize: Herramienta que nos permite generar código YAML de recursos de kubernetes de forma dinámica, utilizada principalmente para segmentar código para distintos ambientes con el concepto de overlays

Instalación de Kubectl

https://kubernetes.io/docs/tasks/tools/install-kubectl-windows/

Durante la instalación del clúster de k3s, kubectl también se encuentra instalado en los nodos del plano de control, por lo que el siguiente procedimiento se puede ejecutar desde la máquina de trabajo del estudiante para poder conectarse de forma remota.

  1. Descargar el binario en un directorio que esté dentro de la variable de entorno %PATH%. Por ejemplo C:\Windows (se requieren privilegios de Administrador). Ejecutar el siguiente comando desde la terminal. Caso contrario se puede utilizar el directorio %HOME% del usuario, por ejemplo: C:\Users\User

curl.exe -LO "https://dl.k8s.io/release/v1.28.4/bin/windows/amd64/kubectl.exe"

Para determinar la última versión estable disponible se puede ejecutar el siguiente comando

curl.exe -sL "https://dl.k8s.io/release/stable.txt"

Instalación de OpenLens

https://www.virtualizationhowto.com/2022/11/openlens-kubernetes-ide-opensource-lens-desktop/

  1. Descargar la herramienta desde el siguiente enlace: https://github.com/MuhammedKalkan/OpenLens/releases

  2. Seleccionar la opción correspondiente al sistema operativo utilizado

  3. Instalar el cliente GUI desde el ejecutable descargado.

Instalación de Helm

En la máquina de trabajo

https://helm.sh/docs/intro/install/

  1. Descargar la versión deseada para el sistema operativo utilizado desde https://github.com/helm/helm/releases

  2. Descomprimir el archivo y copiar el binario helm a un directorio dentro de la variable de entorno %PATH% de la misma manera que se realizó con kubectl

En el control plane

  1. Descargar el paquete con curl

    curl -LO https://get.helm.sh/helm-v3.13.2-linux-amd64.tar.gz

  2. Descomprimir el paquete

    dnf install -y tar
tar -zxvf helm-v3.13.2-linux-amd64.tar.gz

  3. Mover el binario al directorio /usr/local/bin para tenerlo dentro de la variable $PATH

    mv linux-amd64/helm /usr/local/bin/helm

Probar la conexión con el clúster

El comando kubectl puede ser ejecutado tanto desde cualquier máquina control plane así como desde la máquina de trabajo del estudiante.

Procedimiento desde la máquina del estudiante

  1. Descargar el archivos k3s.yml del servidor de control plane. Se encuentra en el directorio /etc/rancher/k3s/k3s.yaml

  2. Modificar la URL de conexión al clúster con un editor de texto, por defecto estará apuntando a server: https://127.0.0.1:6443 y se debe reemplazar por la URL del balanceador de carga. Por ejemplo: server: https://api.k8s.s1.example.com:6443

  3. Se puede copiar este archivo a $HOME/.kube/config, este es el archivo de conexión por defecto utilizado por kubectl

  4. Probar un comando de administración. Ejemplo:

    kubectl get nodes
    Resultado esperado
    [root@m1 ~]# kubectl get nodes
NAME               STATUS   ROLES                       AGE   VERSION
m1.s1.example.com   Ready    control-plane,etcd,master   42d   v1.27.6+k3s1
m2.s1.example.com   Ready    control-plane,etcd,master   42d   v1.27.6+k3s1
m3.s1.example.com   Ready    control-plane,etcd,master   42d   v1.27.6+k3s1
w1.s1.example.com   Ready    <none>                      42d   v1.27.6+k3s1
w2.s1.example.com   Ready    <none>                      42d   v1.27.6+k3s1
[root@m1 ~]#

  5. En el caso de OpenLens, la herramienta permite seleccionar el “contexto” de kubernetes para su respectiva administración. Cada contexto apunta a un clúster específico y un servidor de la API de kuberntes específico, con las credenciales del usuario de sistema system:admin

Re-instalación de k3s con proveedor de red wireguard-native

Note
 Este procedimiento es necesario para poder cambiar el proveedor de red de VXLAN a Wireguard Native. En lugar de realizar una re-instalación, se puede utilizar el archivo config.yaml adecuado desde el inicio.

  1. En todos los nodos master, desinstalar k3s

    /usr/local/bin/k3s-uninstall.sh

  2. En todos los nodos worker, desinstalar k3s

    /usr/local/bin/k3s-agent-uninstall.sh

  3. En el nodo m1, configurarlo e instalarlo

    dnf install -y wireguard-tools
firewall-cmd --permanent --add-port=51820/udp
firewall-cmd --reload
mkdir -p /etc/rancher/k3s
cat <<EOF > /etc/rancher/k3s/config.yaml
tls-san:
  - "api.k8s.s1.example.com"
cluster-init: true
token: secret
flannel-backend: wireguard-native
EOF
curl -sfL https://get.k3s.io | sh -s - server

  4. En los nodos m2 y m3

    dnf install -y wireguard-tools
firewall-cmd --permanent --add-port=51820/udp
firewall-cmd --reload
mkdir -p /etc/rancher/k3s
cat <<EOF > /etc/rancher/k3s/config.yaml
tls-san:
  - "api.k8s.s1.example.com"
server: https://api.k8s.s1.example.com:6443
token: secret
flannel-backend: wireguard-native
EOF
curl -sfL https://get.k3s.io | sh -s - server

  5. En los nodos w1 y w2

    dnf install -y wireguard-tools
firewall-cmd --permanent --add-port=51820/udp
firewall-cmd --reload
mkdir -p /etc/rancher/k3s
cat <<EOF > /etc/rancher/k3s/config.yaml
server: https://api.k8s.s1.example.com:6443
token: secret
EOF
curl -sfL https://get.k3s.io | sh -s - agent
